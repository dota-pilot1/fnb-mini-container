실무 메뉴 저장 구조
==============================================

실무 DB에는 메뉴 데이터가 prog_no 라는 문자열로 계층을 표현해.
미니처럼 parent_id가 아니라 prog_no 숫자 패턴으로 부모-자식 관계 파악.

==============================================
1. DB 테이블: frameone_auth_prog_fsmns
==============================================

  핵심 컬럼:
    prog_no      →  메뉴 고유 번호 (문자열 12자리, 계층 규칙 내장)
    prog_nm      →  메뉴 이름 (예: "브랜드 관리")
    prog_lvl     →  계층 레벨 (3=헤더, 4=사이드바 카테고리, 5=실제 메뉴)
    up_prog_no   →  부모 메뉴 번호
    component    →  컴포넌트 경로 (예: "pages/cps/base/brandmng/BrandRtrv")
    use_yn       →  사용 여부 ('1'=사용, '0'=미사용)
    menu_yn      →  메뉴 표시 여부

==============================================
2. prog_no 규칙 (계층 표현)
==============================================

  prog_no는 12자리 문자열로 앞자리 패턴으로 부모-자식 관계를 나타냄:

  Level 3 (헤더):     061200000000  ← 앞 4자리 '0612', 나머지 0
  Level 4 (카테고리): 061201000000  ← 앞 6자리 '061201', 나머지 0
  Level 5 (메뉴):     061201010000  ← 앞 8자리 '06120101', 나머지 0

  부모 찾는 방법:
    061201010000 (Level 5)
      → 앞 6자리 '061201' + '000000' = 061201000000 (Level 4 부모)

    061201000000 (Level 4)
      → 앞 4자리 '0612' + '00000000' = 061200000000 (Level 3 부모)

==============================================
3. 실제 데이터 예시
==============================================

  prog_no        | prog_lvl | prog_nm       | up_prog_no   | component
  ---------------+----------+---------------+--------------+---------------------
  061200000000   | 3        | 브랜드 관리    | NULL         | NULL
  061201000000   | 4        | 브랜드 등록    | 061200000000 | NULL
  061201010000   | 5        | 브랜드 조회    | 061201000000 | pages/.../BrandRtrv
  061201020000   | 5        | 브랜드 등록    | 061201000000 | pages/.../BrandReg

  트리 구조:
    061200000000 (브랜드 관리)          ← Level 3, 헤더에 표시
      └─ 061201000000 (브랜드 등록)     ← Level 4, 사이드바 카테고리
           ├─ 061201010000 (브랜드 조회) ← Level 5, 실제 메뉴
           └─ 061201020000 (브랜드 등록) ← Level 5, 실제 메뉴

==============================================
4. 미니와의 차이
==============================================

  미니 (parent_id 방식):
    id  parent_id  depth  menu_name
    1   NULL       3      기준 정보 관리
    10  1          4      브랜드
    100 10         5      브랜드 관리

    → parent_id로 직접 부모 참조 (인접 목록 모델)

  실무 (prog_no 문자열 패턴):
    prog_no        prog_lvl  prog_nm
    061200000000   3         브랜드 관리
    061201000000   4         브랜드 등록
    061201010000   5         브랜드 조회

    → prog_no 앞자리 패턴으로 부모 추론
    → up_prog_no 컬럼도 있지만 prog_no 규칙이 우선

==============================================
5. 왜 이런 방식을 쓰나?
==============================================

  장점:
    - prog_no만 보고 계층 파악 가능
    - 정렬 시 prog_no 오름차순 = 계층 순서
    - 범위 검색 가능 (061201로 시작하는 것 = 특정 카테고리 전체)

  단점:
    - 번호 부여 규칙이 복잡
    - 중간에 메뉴 추가 시 번호 재배치 어려움
    - 숫자가 다 차면 확장 불가 (예: 06120199까지 차면 더 못 넣음)

  → 레거시 시스템에서 흔한 방식. 요즘엔 parent_id 방식(미니)이 더 나음.
